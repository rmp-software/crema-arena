<project_specification>
  <project_name>Crema Arena — Specialty Coffee Competition Platform</project_name>

  <overview>
    Crema Arena is an MVP platform for organizing and broadcasting specialty coffee competitions in Brazil —
    latte art TNTs, barista championships, and bar battles. The goal is to validate the idea with minimal
    complexity: an authenticated organizer dashboard, a live display for TV screens, and an audience
    companion screen accessible via QR code.

    Judging model: judge selection, NOT a 0-100 score. In each duel, N judges (default: 3) each pick
    the side they preferred. Final score: "2 × 1" (common win) or "3 × 0" (unanimous). No decimals,
    no averages, no scoring sliders.

    All code, database fields, API routes, and variable names must be in English.
    All user-facing UI text must be in Brazilian Portuguese (pt-BR).

    Bracket format: single elimination. Bracket size is rounded up to the nearest power of 2 —
    unfilled slots receive automatic byes (advance without playing).
  </overview>

  <core_features>

    <auth>
      - Login via email + password (NextAuth CredentialsProvider)
      - Two roles: admin | organizer (role field on the organizers table)
      - Admin: full access including CRUD of other organizer accounts
      - Organizer: manages competitors (global pool) and their own events
      - Middleware protects all /dashboard/* routes
      - Login page at /login with redirect to /dashboard after authentication
      - Initial seed creates a default admin user (ADMIN_EMAIL + ADMIN_PASSWORD from env)
      - No public registration — only admins create new organizer accounts
    </auth>

    <admin_dashboard>
      <organizer_management>
        - Visible only to role=admin
        - List organizers (name, email, created at, role)
        - Create organizer (name, email, password, role: admin|organizer)
        - Edit organizer (name, email, optional password reset, role)
        - Delete organizer (with confirmation modal, cannot delete self)
      </organizer_management>

      <competitor_management>
        - Global competitor pool — reused across multiple events
        - List competitors with search by name or coffee shop
        - Create competitor: name (required), photo (image upload, stored in /public/uploads/
          or Vercel Blob), coffee shop (required)
        - Edit competitor
        - Delete competitor (check for active event participation before deleting)
        - Photo is required — displayed in duel cards on the live display and companion screen
      </competitor_management>

      <event_management>
        - List the organizer's events with status badge (setup | running | finished)
        - Create event: name, date, location (optional), description (optional),
          judges count (default: 3)
        - On creation, event starts with status=setup
        - Event detail page in setup status:
            * Add competitors to the event: search global pool, assign optional seed (integer >= 1)
            * Remove competitors (only in setup)
            * View list of registered competitors with their seeds
            * "Generate bracket" button: calculates bracket_size = next power of 2 >= number
              of registered competitors, places seeds at standard positions, fills remaining
              slots randomly for unseeded competitors, creates byes for empty slots
            * View generated bracket before starting
            * "Start event" button: changes status to running, activates the first round of duels
        - Event detail page in running status:
            * Main panel: current round, list of duels with status (pending/in_progress/completed)
            * "Start duel" button on each pending duel → opens vote capture interface
            * Vote capture interface (tap-to-tally):
                - Shows duel number and round
                - "Fotografar copos" button at the top: uses
                  &lt;input type="file" accept="image/*" capture="environment"&gt; to open
                  the native camera on mobile; photo uploads immediately via
                  POST /api/duels/:id/photo and live display updates on next poll.
                  Button changes to a thumbnail + "Refotografar" once a photo exists.
                  Photo is optional — voting can proceed without it.
                - Two large buttons labeled with each competitor's name
                - Vote counter per side (starts at 0)
                - Total votes required = event's judges_count
                - "Close duel" button active only when all votes are recorded
                - Confirmation modal before closing: displays final score and winner's name,
                  asks for confirmation ("2 × 1 para [Nome]. Confirmar?")
                - On confirm: winner_entry_id saved, duel marked completed, next bracket duel
                  updated with the winner
            * Wildcard: when a competitor drops out before their next duel:
                - "Wildcard" button on the affected pending duel
                - Three options presented to the organizer:
                    1. Walkover — opponent advances automatically without playing
                    2. Pick eliminated — scrollable list of eliminated competitors in the event,
                       organizer selects one manually
                    3. Random pick — system randomly selects from eliminated competitors,
                       shows selected name and photo for confirmation
                - Original competitor marked as dropped_out
            * Full bracket view with progress
            * "Close event" button when all final duels are complete (status → finished)
        - Event detail page in finished status:
            * Final bracket with winner highlighted and trophy mark
            * Final leaderboard
        - Links section (visible in running and finished status):
            * "Live Display" — "Copy URL" button that copies /live/[eventId] to clipboard
            * "Audience (QR)" — displays server-generated QR code linking to /e/[eventId]
              and a button to download the QR as PNG
      </event_management>
    </admin_dashboard>

    <live_display>
      - Route: /live/[eventId] — public, unauthenticated, optimized for large screens (TV 1920×1080)
      - Auto-refresh via polling every 5 seconds (fetch in useEffect with setInterval)
      - Full-screen layout, espresso background (--bg-inverse), no nav, no user interaction
      - Content when event is running:
          * Central focus: active duel — when pour_photo_url is set, display the pour photo
            large and centered (the actual latte art photo taken by the organizer), with
            competitor names and coffee shops as overlaid captions at the bottom of the image.
            When no pour photo exists yet, fall back to the two competitor profile cards
            side by side with photo, name, and coffee shop.
          * Duel score in large Geist Mono ("2 × 1")
          * LIVE status (mint dot + "AO VIVO" mono caps label)
          * Simplified mini bracket of the current round below or to the side
          * Next duel (if any) displayed with less prominence
          * QR code in the bottom-right corner linking to /e/[eventId]
            with label "Acompanhe pelo celular"
      - Content when event is finished:
          * Full-screen podium UI showing the top 5 competitors:
              - Ranking determined by elimination round (later round = higher rank),
                with total votes received across the tournament as tiebreaker
              - Classic podium layout: 2nd left, 1st center (tallest), 3rd right,
                4th and 5th flanking lower on each side
              - Each podium slot shows: competitor photo (large, circular), name,
                coffee shop, position number, and score summary (e.g. "3 vitórias · 8 votos")
              - 1st place slot uses --gold accent, trophy.svg mark, and "CAMPEÃO" stamp seal
              - 2nd and 3rd slots use --fg-2 / --brand-soft treatment
              - 4th and 5th slots smaller, same card style but less prominent
              - Espresso background (--bg-inverse) with crema text (--fg-inverse)
              - Event name and date displayed at the top in Bricolage Grotesque
          * If the bracket has fewer than 5 competitors total, show only as many
            positions as there are eliminated competitors (minimum: just 1st place)
      - Content when event is in setup (not yet started):
          * Event name, date, "O evento ainda não começou"
      - No interactive elements except auto-refresh
    </live_display>

    <live_companion>
      - Route: /e/[eventId] — public, unauthenticated, mobile-first
      - Same URL before, during, and after the event — content changes based on status
      - Auto-refresh via polling every 5 seconds
      - Mobile-first layout with sticky header (event name + status badge)
      - Tab navigation: Ao vivo | Chave | Leaderboard
      - "Ao vivo" tab:
          * Current duel: if pour_photo_url is set, show the pour photo prominently with
            competitor names as captions; otherwise show competitor profile cards.
          * Partial score in Geist Mono
          * "Próximo duelo" section below when available
          * When no duel is in progress: "Aguardando próximo duelo..."
      - "Chave" tab:
          * Scrollable full bracket view
          * Each duel shows: competitors, final score (if completed), winner badge
          * Byes labeled as such with no extra detail
          * Wildcard entries with "Wildcard" badge
      - "Leaderboard" tab:
          * During event: partial standings (winners by round)
          * After close: full final leaderboard with position, name, coffee shop,
            duels won, and total votes received in the tournament
      - Content when setup: event name, date, list of registered competitors
      - Content when finished: focus on leaderboard + final bracket, no "Ao vivo" tab
    </live_companion>

    <tables>
      <organizers>
        - id (uuid, PK)
        - name (text)
        - email (text, unique)
        - password_hash (text)
        - role (enum: admin | organizer, default organizer)
        - created_at, updated_at
      </organizers>

      <competitors>
        - id (uuid, PK)
        - name (text)
        - photo_url (text) — path in /public/uploads/ or Vercel Blob URL
        - coffee_shop (text)
        - created_by (uuid, FK organizers.id)
        - created_at, updated_at
      </competitors>

      <events>
        - id (uuid, PK)
        - organizer_id (uuid, FK organizers.id)
        - name (text)
        - date (date)
        - location (text, nullable)
        - description (text, nullable)
        - judges_count (int, default 3)
        - status (enum: setup | running | finished, default setup)
        - bracket_size (int, nullable) — power of 2, calculated on bracket generation
        - created_at, updated_at
      </events>

      <event_entries>
        - id (uuid, PK)
        - event_id (uuid, FK events.id)
        - competitor_id (uuid, FK competitors.id)
        - seed (int, nullable) — bracket seed position
        - status (enum: active | eliminated | wildcard_in | dropped_out, default active)
        - eliminated_at_round (int, nullable) — round in which they were eliminated
        - created_at
        - UNIQUE(event_id, competitor_id)
      </event_entries>

      <duels>
        - id (uuid, PK)
        - event_id (uuid, FK events.id)
        - round (int) — round number (1 = first round, increases each phase)
        - position (int) — position within the round (1-based)
        - entry_a_id (uuid, FK event_entries.id, nullable) — null = bye
        - entry_b_id (uuid, FK event_entries.id, nullable) — null = bye
        - votes_a (int, default 0)
        - votes_b (int, default 0)
        - winner_entry_id (uuid, FK event_entries.id, nullable)
        - status (enum: pending | in_progress | completed | walkover, default pending)
        - wildcard_type (enum: walkover | manual | random, nullable)
        - next_duel_id (uuid, FK duels.id, nullable) — destination duel for the winner
        - next_duel_slot (enum: a | b, nullable) — which slot the winner fills in the next duel
        - pour_photo_url (text, nullable) — photo of the two cups taken by the organizer during the duel
        - completed_at (timestamp, nullable)
        - created_at
      </duels>

      <votes>
        - id (uuid, PK)
        - duel_id (uuid, FK duels.id)
        - side (enum: a | b)
        - created_at
      </votes>
    </tables>
  </database_schema>

      <!-- Admin only -->
      - GET    /api/organizers           — list all organizers
      - POST   /api/organizers           — create organizer
      - PUT    /api/organizers/:id       — edit organizer
      - DELETE /api/organizers/:id       — delete organizer
    </organizers>

    <competitors>
      - GET    /api/competitors           — list with search (?q=)
      - POST   /api/competitors           — create (with photo upload)
      - PUT    /api/competitors/:id       — edit
      - DELETE /api/competitors/:id       — delete
    </competitors>

    <events>
      - GET    /api/events                — list organizer's events
      - POST   /api/events                — create event
      - GET    /api/events/:id            — event detail
      - PUT    /api/events/:id            — edit event
      - DELETE /api/events/:id            — delete event (setup only)

      - POST   /api/events/:id/entries          — add competitor to event
      - DELETE /api/events/:id/entries/:entryId — remove competitor (setup only)

      - POST   /api/events/:id/bracket/generate — generate bracket (setup only)
      - POST   /api/events/:id/start            — start event (setup → running)
      - POST   /api/events/:id/finish           — close event (running → finished)

      - GET    /api/events/:id/bracket          — full bracket (public)
      - GET    /api/events/:id/current-duel     — active duel (public)
      - GET    /api/events/:id/leaderboard      — standings (public)
      - GET    /api/events/:id/qr               — QR code as PNG (public)
    </events>

    <duels>
      - POST   /api/duels/:id/start     — start duel (→ in_progress)
      - POST   /api/duels/:id/vote      — record vote { side: "a" | "b" }
      - POST   /api/duels/:id/complete  — close duel { winner: "a" | "b" }
      - POST   /api/duels/:id/wildcard  — apply wildcard { type: "walkover"|"manual"|"random", entry_id? }
      - POST   /api/duels/:id/photo    — upload pour photo (multipart/form-data); saves to storage,
                                         updates pour_photo_url, immediately visible on live display
    </duels>
  </api_endpoints_summary>

  <key_interactions>
    <bracket_generation>
      1. Organizer adds competitors to the event with optional seeds
      2. Clicks "Gerar chave"
      3. System calculates bracket_size = next power of 2 >= total registered competitors
      4. Seeds are placed at standard single-elimination positions (1 at top, 2 on opposite side, etc.)
      5. Unseeded competitors fill remaining slots in random order
      6. Remaining slots (bracket_size - total_competitors) become byes
      7. System creates duel records for round 1 and cascades next_duel_id + next_duel_slot
         through the entire bracket tree upfront
      8. Byes are resolved automatically: the competitor advances without a played duel
      9. Organizer reviews the generated bracket, then clicks "Iniciar evento"
         (no bracket editing after generation)
    </bracket_generation>

    <running_a_duel>
      1. Admin panel shows current round with pending duels
      2. Organizer clicks "Iniciar duelo" — status changes to in_progress
      3. Tap-to-tally interface opens with a "Fotografar copos" button at the top.
         Organizer points their phone at the table and taps it — native camera opens,
         photo is taken, uploaded immediately. Live display switches from competitor
         profile cards to the pour photo within the next poll cycle (~5s).
         Button becomes a thumbnail + "Refotografar" if they want to retake.
      4. Two large buttons labeled with each competitor's name — each tap records a vote
         and increments the on-screen counter (photo and voting are independent)
      5. When all judges_count votes are recorded: "Encerrar duelo" button activates
      6. Confirmation modal displays the final score and winner's name:
         "2 × 1 para [Nome]. Confirmar?" — no name mapping needed
      7. On confirm: winner_entry_id saved, loser marked as eliminated, next duel in the
         bracket updated with the winner, live display and companion update on next poll
    </running_a_duel>

    <wildcard_flow>
      1. Competitor drops out before their next scheduled duel
      2. Organizer opens the affected pending duel and clicks "Wildcard"
      3. Modal presents three options:
         a. Walkover — opponent advances automatically, duel marked as walkover
         b. Pick eliminated — paginated list of competitors with status=eliminated
            in that event; organizer selects one manually
         c. Random pick — system randomly selects from eliminated competitors,
            displays selected name and photo for confirmation ("Confirmar ou sortear novamente")
      4. Original competitor marked as dropped_out
      5. If wildcard (b or c): new event_entry created with status=wildcard_in,
         duel updated with new entry
      6. If walkover (a): duel marked completed/walkover with opponent as winner
    </wildcard_flow>

    <live_surface_updates>
      - Live display (/live/[eventId]): useEffect with setInterval 5000ms fetching
        /api/events/:id/current-duel and /api/events/:id/bracket — no WebSocket, no SSE
      - Live companion (/e/[eventId]): same 5-second polling pattern
      - No real-time optimization needed for MVP — polling is sufficient
    </live_surface_updates>
  </key_interactions>
